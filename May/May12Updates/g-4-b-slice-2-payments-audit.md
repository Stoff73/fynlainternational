# G-4-b — Slice 2: Revolut Webhook + Payment Endpoints

## Executive summary

The Revolut webhook surface is the highest-risk endpoint family in the application — successful exploitation could trigger subscription activations without real payment capture, or replay activations to drain a Revolut wallet via grant of premium access. The HMAC-SHA256 signature verification (`RevolutService::verifyWebhookSignature`) is correctly implemented — algorithm matches Revolut docs, `hash_equals` used for constant-time comparison, 5-minute timestamp tolerance applied **before** signature hashing (good — avoids timing-attack surface on signed payloads), multi-signature support handles key rotation. The webhook → DB write path correctly uses `lockForUpdate()` + `status === 'completed'` short-circuit for idempotency, the AwinConversionJob is idempotent via `awin_fired_at`, and merchant_ref cross-references are computed.

However, **four HIGH-severity issues** require fixes before slice 2 exits: (1) `payments.revolut_order_id` has **no unique database constraint** — the idempotency guarantee depends entirely on row-level `lockForUpdate`, but two simultaneous webhook deliveries that arrive before either row exists will both `Payment::create()` with the same order ID; (2) `PaymentController::confirmPayment` accepts `processing` and `pending` order states as activation triggers, letting users activate their subscription **before money is captured** — a user can submit a payment that never completes (or charge-back immediately) and retain a full billing cycle of premium access; (3) `WebhookController::handleOrderCompleted` cross-references `merchant_order_ext_ref` against the payment ID but **only logs a warning** on mismatch — it then proceeds to activate the subscription anyway, defeating the only protection against confused-deputy attacks; (4) **payment endpoints do not require MFA** — `cancelSubscription`, `upgradeSubscription`, `deleteAllData`, `confirmPayment` are all `auth:sanctum` only, so a stolen session token (without the second factor) can charge, cancel, or destroy user data.

**Eight MEDIUM-severity items** concern defense-in-depth weaknesses: webhook handler swallows exceptions and returns 200 (Revolut never retries failed processing); no idempotency on the upgrade flow (concurrent tabs create duplicate Revolut orders + Payment rows); Revolut HTTP calls have **no `->timeout()`** (overlaps with G-5 H-3 but applies here too); `handleOrderCompleted` always resets `current_period_start = now()` even for renewal cycles; the webhook signature verifier has **zero test coverage**; missing `capture_mode` defaults to `'automatic'` silently; unhandled webhook events log at `info` level only (silent for future Revolut API changes); upgrade flow uses `revolut_order_id = 'pending'` placeholder which collides with itself across concurrent requests.

**Five LOW-severity items** logged for triage. `SubscriptionRenewalService::handleRenewalPayment` is **dead code** (never called from the webhook handler or any scheduled job — only the cancelled/overdue/finished handlers are wired in). The webhook timestamp tolerance is 5 min (could be tighter at 2 min). Webhook secret-missing log is silent rather than actionable.

## Recommended fix order (if all HIGH+MEDIUM applied this session)

Estimated ~2.5 hours for HIGH; ~5 hours for HIGH+MEDIUM. H-1 + M-2 + M-8 share a unique-index migration so batch them.

Migration must (a) deduplicate any existing duplicates first by keeping the latest `completed` row, (b) backfill the `'pending'` placeholder for upgrade flow to a per-row unique placeholder (`"upgrade_{$id}"`), (c) add the unique index. Update `WebhookController::handleOrderCompleted` to catch `QueryException`-23xxx and treat as idempotent. Update `upgradeSubscription` to use a unique placeholder. 60 min including migration + rollback test.
2. **H-2** — Tighten `confirmPayment` acceptable states to match the webhook handler. 15 min, single line + test.

4. **H-4** — Apply `mfa.verified` middleware to the authenticated payment route group. Audit existing mobile/iOS token flow to ensure `mfa_verified` ability is granted at biometric-login time. 45 min + test that MFA-enabled user without verification gets 403 on each endpoint.
5. **M-1** — Webhook 500-on-handler-failure so Revolut retries. 15 min + test.
6. **M-3** — `->timeout(10)->connectTimeout(5)` on every Revolut HTTP call. 30 min (sweep `RevolutService.php` + `RevolutSubscriptionService.php`). Pairs with the broader G-5 H-3 timeout sweep.

9. **M-6** — Fail-loud on missing `capture_mode`. 15 min + test.
