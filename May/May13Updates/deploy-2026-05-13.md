## Behaviour changes to expect on dev

### Slice 1 (auth)
- Password-reset response shape now uniform regardless of whether email exists (no enumeration leak).
- MFA disable + recovery-code regen now require a current TOTP/recovery code in addition to password.
- Admin auto-promote now happens AFTER `Auth::attempt` (no writes on failed login) and writes `ACTION_ADMIN_PROMOTED` audit log entries.
- Password change rotates all OTHER tokens (current session preserved).
- MFA verify scopes token deletion to `name='auth_token'` (mobile/preview tokens survive web MFA).
- Spouse account creation requires explicit email (no more `@temp.fps.com` synthesised emails).

### Slice 2 (payments)
- Concurrent webhook deliveries for the same `order_id` now safely no-op via DB unique index + `QueryException` catch.
- `POST /api/payment/confirm` now returns 400 when Revolut order state is `processing` or `pending` (was 200 + activation). Users must wait for actual capture.
- `POST /api/payment/confirm` now returns 400 if Revolut omits `capture_mode` (was silently treated as automatic).
- Webhook handler refuses to activate on `merchant_ref` mismatch (was warning-only + activated anyway).
- Payment write endpoints (`/create-order`, `/confirm`, `/upgrade`, `/cancel-subscription`, `/delete-all-data`) now return 403 for MFA-enabled users without an MFA-verified token. Read endpoints (`/trial-status`, `/billing-history`, `/validate-discount`, invoice download) unchanged.
- Concurrent `/api/payment/upgrade` requests for the same user now return 409 (`Cache::lock` collision).
- Webhook returns 5xx (Revolut retries) on handler exceptions, was 200 (lost activations).
- Renewal cycles preserve period continuity: new `current_period_start = old current_period_end` rather than `now()`.
- All Revolut outbound HTTP calls now have `timeout(10)->connectTimeout(5)`.

## Rollback

If a regression appears:

```bash
# Rollback the new migration (single migration; safe one-step)
php artisan migrate:rollback --step=1 --force

# Then upload prior PHP files via File Manager (Site Tools → File Manager → version history if available)
# OR git pull to the previous tag if a tag was cut pre-deploy.
```

The migration's `down()` drops the unique index but does NOT undo the placeholder backfills (`upgrade_pending_…`, `deduped_…`). Those new placeholders are inert — they don't affect runtime — so no rollback action is needed for them.
